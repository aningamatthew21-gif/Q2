1. Executive Summary

The MIDSA Quote App enforces access via a role-based access-control (RBAC) catalogue defined in a single file (shared/permissions.js) that is imported by both the React frontend (to hide UI) and the Express backend (to gate routes). This eliminates the most common cause of authorisation drift — the UI showing a button while the server forgets to gate the corresponding API call.

Three layers of defence apply, in order, on every privileged action:

LayerMechanismWhat it asksFailure mode
1 JWT authentication "Is this request signed by a token we issued, and is it still valid (≤ 15 min access token)?" 401 Unauthorized
2 Role × Action grant (RBAC) "Does this role hold the action key required by this route?" 403 Forbidden
3 Separation-of-Duties predicate "For approval-class actions, is the approver a different person than the creator?" 403 with SoD-specific message

Beyond these three, an immutable audit log (QA_AUDIT_LOGS) records every mutation with the actor's email, before/after diff, request ID and IP. This is the compensating control that ISO 27001 A.5.3 explicitly accepts when role-based SoD bypasses are unavoidable for operational reasons (see §5).

Why this design satisfies A.5.3 (Segregation of Duties): the four approval-class actions (invoice.approve.sales, invoice.approve.finance, rfq.approve.award, invoice.customer_action) all enforce "approver ≠ creator/sender" at the route level. Where role-level authority overrides this for operational reasons (admin / finance_head), the override is documented in code, flagged in this document, and observable in the audit log.

4. Per-Role Capability Detail

Administrator System tier

Who: IT lead, app owner, designated incident responder. Typically 1–2 people for the whole organisation.

The single role with universal grant. Holds every action key in ACTIONS. Used for provisioning users, debugging via "act-as," emergency override of business rules, and configuration changes that span departments.

Privileged — admin-only
  • 👤 user.manage — provision / promote / deactivate users
  • 🎭 user.impersonate — act as another user (every action logged with the real actor)
  • 🚨 system.act_as_emergency — emergency override of SoD / business rules (double-audited)
  • 🛰 system.errors.read — open the real-time Error Monitor (QA_ERROR_LOG observability + ack/resolve workflow)
Universal grant
  • Every action listed in §3 — admin's row is fully checked.
  • SoD bypass applies on the three approval-class actions where it's documented (sales-approve, finance-approve, customer-action).
What admin should NOT do day-to-day: originate quotes, log payments, or perform approvals in normal operation. Every admin action is audit-logged with the actor email; using admin for routine business actions makes the audit signal noisy and risks an ISO 27001 A.5.18 finding (least privilege). The right pattern is: grant the user the correct tiered role; reserve admin for cases where no tiered role fits.

5. Separation-of-Duties Rules

Four runtime invariants are enforced on the approval-class actions. They run after the RBAC permission check passes, so they answer the question: "Even though this role has the permission, is the specific actor the wrong person to do this on this specific record?"

ISO/IEC 27001:2022 A.5.3 — Segregation of Duties. Where it is operationally infeasible to enforce strict separation (e.g. small teams, off-hours, single-administrator firms), A.5.3 permits role-based overrides provided compensating controls exist. In this system, the compensating control is the immutable audit log (QA_AUDIT_LOGS) which captures actor email, before/after, request ID, and IP for every privileged action. Two of the four SoD rules carry a documented role-based bypass for admin and finance_head; this is intentional and matches the standard's accepted pattern.

SoD-1 · invoice.approve.sales

Where it fires: when an invoice transitions Pending Approval → Approved on the sales side.

Predicate:

approver.email ≠ invoice.createdBy AND approver.email ≠ invoice.salesPersonId

Intent: the user who drafted or owned the quote cannot also approve it. Forces a second pair of eyes between authoring and sending.

Bypass: admin and finance_head roles pass automatically. Justified by A.5.3 — these roles carry organisational signing authority; the audit log is the compensating control.

SoD-2 · invoice.approve.finance

Where it fires: when an invoice transitions through finance-side approval (controller signature).

Predicate:

approver.email ≠ invoice.createdBy AND approver.email ≠ invoice.salesPersonId

Intent: mirrors SoD-1 — the original sales author cannot also stamp the controller signature. Forces clean three-step separation: author → sales-head approve → finance-head approve.

Bypass: admin and finance_head roles pass automatically. Same rationale as SoD-1.

SoD-3 · rfq.approve.award

Where it fires: when a procurement head clicks "Approve award" on an RFQ that an officer recommended.

Predicate:

approver.email ≠ rfq.recommendedBy

Intent: the procurement officer who recommended cannot also approve. Forces a second person before money is committed.

Bypass (added 2026-05-26): admin, procurement_head (domain authority), and finance_head (parity with the other 3 SoDs + cross-departmental spend oversight) pass automatically. Justified under ISO/IEC 27001:2022 A.5.3 — the audit log (QA_AUDIT_LOGS) records actor email + before/after for every state change, providing the compensating control the standard requires. Closes the last admin-bypass gap so admins truly have no SoD restrictions anywhere in the system.

SoD-4 · invoice.customer_action

Where it fires: when an internal user clicks "Mark accepted" or "Mark rejected" on a customer's behalf (e.g. customer confirmed by phone/email).

Predicate:

actor.email ≠ invoice.sentBy

Intent: the sales rep who sent the invoice cannot also pretend the customer said yes. Forces a second internal party to confirm the customer's voice — a guard against bookkeeping fraud where a rep marks an unsent invoice "Accepted" to close their pipeline.

Bypass (added 2026-05-26): admin and finance_head pass automatically. Justified by A.5.3 — when a customer confirms by phone/email and the head/admin records it, the audit log is the compensating control. Previously this rule was the only one without the bypass, which caused legitimate admin actions to silently fail.

6.4 Escalation paths

When an officer hits a permission boundary, the right next step:

  • Sales officer needs to approve their own quote → send to Sales Head (SoD-1 will block self-approval anyway).
  • Sales officer needs the customer-action recorded on an invoice they sent → ask another sales rep, sales head, finance head, or admin (SoD-4 blocks self-action; the latter two bypass SoD as a compensating control).
  • Procurement officer needs to approve an award they recommended → send to Procurement Head or admin (admin / procurement-head / finance-head all bypass SoD-3 with audit-log compensation).
  • Finance officer needs to reverse a payment older than 24h → send to Finance Head (server-side window enforced in collections.js).
  • Any officer needs to edit procurement thresholds → send to Procurement Head or Finance Head (both can edit).

7. Customer Portal & External Access

Customers authenticate with the same OTP flow as internal users but receive a JWT whose role is customer. Their access surface is intentionally minimal:

  • Page-level: only customerPortal renders for them; every other page returns Forbidden.
  • Record-level: the backend's GET /api/customers/:id handler additionally checks that the customer's email matches the customer record being requested. A crafted ID in the URL cannot escalate to another customer.
  • Mutation surface: the customer can only flip an invoice in Awaiting Acceptance state to Customer Accepted or Customer Rejected. They cannot edit amounts, line items, or any other invoice field.
  • Audit: every customer action is logged with the customer's email and IP, identically to internal user actions.

Co-browse exception: customerPortal is the only page with PAGE_PERMISSIONS[page] = null (public for any authenticated user). This lets internal support staff open the portal to debug a customer issue. The customer-scope check on the API still applies, so internal users only see the customer record they navigated to — they cannot list other customers from inside the portal.

8. Outside the Model

Three deliberate omissions you should know about, with their justification:

CapabilityWhy not gatedCompensating control
Login page (login) Anyone reaching the app can attempt to log in — that's the point of an authentication flow. Rate-limited OTP, HMAC-SHA256 hashed OTPs, 200ms minimum latency floor against enumeration.
Customer portal page-level (customerPortal) Internal staff need to co-browse for support purposes. Record-level check on the customer API restricts a customer role to their own record.
Audit-log writes (POST /api/audit-logs) Every authenticated user fires fire-and-forget telemetry. Gating it would silently break activity logging on permission-edge cases. The endpoint validates the actor against the JWT — a user cannot log activity as someone else.

9. Audit Trail & Compensating Controls

Two complementary audit mechanisms run in parallel:

  1. auditMiddleware (backend, automatic) — runs after every POST/PUT/PATCH/DELETE. Writes a row to QA_AUDIT_LOGS containing: actor email, role, request method + path, request body (PII-masked: passwords, OTPs, tokens replaced with [REDACTED]), response status, request ID (for cross-correlation with structured logs), client IP, user-agent, and timestamp. Cannot be bypassed by application code.
  2. logActivity() (frontend, semantic) — explicit calls from React components on meaningful events (page views, button clicks the auto-middleware can't infer). Validated server-side against the JWT.

These two together satisfy ISO/IEC 27001:2022 A.8.15 (Logging) and A.5.34 (Privacy & protection of PII), and serve as the compensating control that A.5.3 requires when role-based SoD bypasses are accepted (see §5).
